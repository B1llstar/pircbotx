 /** A very fast and memory efficient class to encode and decode to and from BASE64 in full accordance
 6   
  * with RFC 2045.<br><br>
 7   
  * On Windows XP sp1 with 1.4.2_04 and later ;), this encoder and decoder is about 10 times faster
 8   
  * on small arrays (10 - 1000 bytes) and 2-3 times as fast on larger arrays (10000 - 1000000 bytes)
 9   
  * compared to
 10   
  * <code>sun.misc.Encoder()/Decoder()</code>.<br><br>
 11   
  *
 12   
  * On byte arrays the encoder is about 20% faster than Jakarta Commons Base64 Codec for encode and
 13   
  * about 50% faster for decoding large arrays. This implementation is about twice as fast on very small
 14   
  * arrays (&lt 30 bytes). If source/destination is a
 15   
  * <code>String</code> this
 16   
  * version is about three times as fast due to the fact that the Commons Codec result has to be recoded
 17   
  * to a
 18   
  * <code>String</code> from
 19   
  * <code>byte[]</code>, which is very expensive.<br><br>
 20   
  *
 21   
  * This encode/decode algorithm doesn't create any temporary arrays as many other codecs do, it only
 22   
  * allocates the resulting array. This produces less garbage and it is possible to handle arrays twice
 23   
  * as large as algorithms that create a temporary array. (E.g. Jakarta Commons Codec). It is unknown
 24   
  * whether Sun's
 25   
  * <code>sun.misc.Encoder()/Decoder()</code> produce temporary arrays but since performance
 26   
  * is quite low it probably does.<br><br>
 27   
  *
 28   
  * The encoder produces the same output as the Sun one except that the Sun's encoder appends
 29   
  * a trailing line separator if the last character isn't a pad. Unclear why but it only adds to the
 30   
  * length and is probably a side effect. Both are in conformance with RFC 2045 though.<br>
 31   
  * Commons codec seem to always att a trailing line separator.<br><br>
 32   
  *
 33   
  * <b>Note!</b>
 34   
  * The encode/decode method pairs (types) come in three versions with the <b>exact</b> same algorithm and
 35   
  * thus a lot of code redundancy. This is to not create any temporary arrays for transcoding to/from different
 36   
  * format types. The methods not used can simply be commented out.<br><br>
 37   
  *
 38   
  * There is also a "fast" version of all decode methods that works the same way as the normal ones, but
 39   
  * har a few demands on the decoded input. Normally though, these fast verions should be used if the source if
 40   
  * the input is known and it hasn't bee tampered with.<br><br>
